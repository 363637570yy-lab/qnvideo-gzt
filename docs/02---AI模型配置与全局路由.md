# AI模型配置与全局路由

最近更新：2026-06-10

## 功能定位

AI 配置是系统的模型能力中心，负责管理文本、图像、视频、音频等服务商配置，并为不同业务场景提供可控、可切换、可降级的模型路由。

## 当前项目情况

- AI 配置入口是 `/ai-config`，仅管理员可访问。
- 配置主表是 `ai_service_configs`，业务场景映射表是 `ai_model_map`。
- 管理页已分为 AI 配置、提示词高级设置、业务场景高级设置、运行观测、SD2 资产管理。
- 当前支持文本/对话、图像、视频、TTS、即梦2角色认证等配置类型。
- 当前已支持全局路由策略：优先级、并发上限、请求超时、视频等待、重试次数、冷却时间。
- 当前已新增 `ai-runtime` 轻量运行时门面：候选选择、业务场景映射、冷却/鉴权失败跳过、重试和超时读取逐步统一到同一入口。
- 当前已新增模型运行观测页：管理员可按服务类型、状态和时间范围查看模型调用摘要、配置/模型 Top 和最近脱敏调用日志。
- 当前已新增模型能力标签：系统根据服务类型、接口规范、provider 和模型名推断是否支持参考图、多参考图、首尾帧、局部预览、异步任务、TTS、用量返回等能力，并在 AI 配置表展示。
- 当前已新增统一业务场景 key 定义：故事、角色/场景/道具提取，分镜提示词，角色/场景/道具/分镜图片，TTS 和分镜视频均由后端统一给出定义，前端“业务场景高级设置”动态读取。
- 接口规范只负责协议适配，不重复管理超时、重试、并发；这些运行控制统一由全局路由策略负责。
- 单个 API 配置/配置内模型的每日用量限制不属于全局路由策略；当前在 AI 配置列表行内维护，默认 0 表示不限制，详细口径见 `12---模型用量统计与额度治理.md`。
- 当前已新增登录用户可读的“用量中心”，入口在首页和创作页顶部，按文本、图片、视频、音频查看整站用量。
- 全局路由策略必须同时作用于后端执行队列和前端一键流程提交节奏，避免前端批量提交速度与后端真实并发策略脱节。
- 当前视频协议已覆盖 OpenAI 兼容、Sora 中转、Veo3、火山 Seedance、Seedance 全能、DashScope、Vidu、Jimeng 自建等方向。
- 当前图片协议已覆盖 OpenAI 兼容、OpenAI GPT Image、CLIProxyAPI gpt-image-2、火山 Seedream、DashScope/Qwen、Gemini、Kling、NanoBanana 等方向；OpenAI GPT Image/CLIProxy 同一条配置内可用“API 接口”切换 Images API 与 Responses API。

## 当前已落地能力

- AI 配置仅管理员可访问。
- 支持文本、图像、视频、音频/TTS 等服务类型。
- 支持多个配置按排序使用。
- 支持运行时公开读取当前模型路由。
- 支持失败计数、冷却、禁用状态、最后错误等字段。
- 支持部分服务商连接测试和素材资产查询。
- 支持通过 `api_protocol` 区分厂商/协议大类，例如普通 OpenAI 兼容、官方 GPT Image、CLIProxyAPI gpt-image-2；OpenAI GPT Image/CLIProxy 内部再用 `settings.image.api_mode` 在 `images` 与 `responses` 间切换，不要求用户重复新建配置。
- OpenAI GPT Image/CLIProxy 图片协议按模型能力控制参数发送：`gpt-image-2` 尺寸归一化到官方有效范围，透明背景和 `input_fidelity` 等易失败字段会跳过；CLIProxy `quality` 默认保守跳过，可通过 `settings.image.send_quality=true` 显式打开。
- OpenAI GPT Image/CLIProxy 配置页已增加 CookSleep 风格的“API 接口”下拉：Images API 走 `/images/generations|/images/edits`，模型默认偏向 `gpt-image-2`；Responses API 走 `/responses` + `image_generation` 工具，主模型默认偏向 `gpt-5.5`，工具模型默认 `gpt-image-2`。
- 制作页一键生成图片/视频时会读取运行时路由策略中的并发配置；后端队列仍是最终保护层。
- 图片、文本 Vision、帧提示词、TTS 等入口已开始写入明确 `scene_key`，模型映射、冷却切换和调用日志能够按业务场景追踪。
- 文本、图片、视频、TTS 调用已进入统一用量账本；视频链路开始写模型调用日志，任务状态接口可返回关联模型诊断摘要。
- 制作页一键动作区已恢复文本、图像、视频、音频的本次生成模型选择框，支持自动路由或指定配置。
- 制作页左侧导航不再承担本次生成模型下拉选择；用户侧只保留顶部选择框和刷新入口，管理员配置入口仍放在顶部栏/管理入口，避免与全局路由策略形成重复控制面。
- AI 配置列表已增加行级今日用量和每日限额入口，可按配置或配置内模型设置日请求、token、图片张数、视频秒数、音频秒数限制。

## 当前问题与不足

- “自动选择模型”的业务规则仍需继续产品化展示：按服务类型、业务场景、项目规格、模型能力还是成本排序。
- 不同模型的能力边界已开始以能力标签展示；后续还需要补人工编辑覆盖、能力筛选和生成入口提醒。
- 视频链路已先拆出协议/URL/模型选择和返回解析基础模块；厂商创建请求、轮询查询、视频落盘仍需继续外移到 `ai-video` adapter。
- 当前配置管理仍偏技术表单，运行观测已先补齐慢请求和失败排查入口，但还缺少面向创作流程的模型能力说明和风险提示。
- OpenAI 类图片中转容易把官方 Images API、Responses API、CLIProxyAPI 兼容层混用；当前已把用户可见层收敛为同配置内“API 接口”切换，后续还需继续补诊断摘要，显示实际请求路径和关键字段是否被发送。
- CLIProxyAPI 文档或实测能力更新较快，不能把第三方文章中的所有字段直接当作官方契约；新增字段必须走接口规范分支、默认值和失败回退策略。

## 方案池 / 待确认讨论

- 引入模型能力标签：文生图、图生图、首尾帧、全能视频、TTS、多参考图、长视频、同步/异步。
- 管理员是否需要在业务场景路由中按“质量/速度/成本”维护多套映射方案。
- 引入成本策略：优先质量、优先速度、优先低成本、备用路线。
- 管理员是否可以为某个项目单独指定模型策略。
- 后续建立 `ai-runtime` 统一执行器：配置选择、重试、冷却、失败切换只保留一份；图片 adapter 已先完成主体拆分，文字、视频继续逐步跟进。
- 文字模型已开始拆为 `ai-text`：先拆 OpenAI 兼容 Chat 的非流式/流式传输与响应解析；后续再拆普通文本、结构化 JSON、流式文本和不同厂商协议 adapter。
- TTS 协议已开始拆为 `ai-audio`：MiniMax 和 OpenAI 兼容 TTS URL 与请求格式独立维护；当前仍保持原同步接口，后续再接入任务队列。

## 验收清单

- 管理员配置多个同类模型后，系统能按明确顺序选择。
- 单个配置失败时，错误能记录，并按规则切换或提示。
- 管理员能在运行观测页看到最近调用、失败摘要、耗时和基础消耗统计。
- 普通用户无法进入 AI 配置页面和管理接口。
- 管理员在业务场景高级设置中看到的场景 key 与后端调用日志中的 `scene_key` 保持一致，不再依赖前端硬编码列表。
