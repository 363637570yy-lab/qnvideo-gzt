# 文件存储生命周期与清理

最近更新：2026-06-09

## 功能定位

文件存储生命周期规范用于管理上传素材、生成图片、生成视频、音频、缩略图、导出文件、缓存、临时文件和回收站，确保数据库只存引用，媒体正文稳定落盘、可追踪、可清理、可恢复。

本文关注底层文件系统、存储路径、引用关系、清理策略和恢复能力。媒体素材库的列表、筛选、预览、选择、业务删除入口和用户可见资源状态放在 `06---媒体素材库与文件资源.md`。

## 存储原则

- 图片、音频、视频、上传文件和生成结果不存入 PostgreSQL 正文。
- 数据库保存文件引用、元数据、归属、来源和引用关系。
- 对外展示优先使用 `/static/<local_path>`。
- 不长期使用 base64 作为前端主展示字段或数据库主引用。
- 服务器 `/opt/ai-video/qnvideo-gzt/data` 是持久化目录，部署、拉代码、容器重建不得覆盖。

## 数据库文件字段

文件相关表或业务表应尽量包含：

- `local_path`：相对 `data/storage` 的路径。
- `url` 或 `image_url`：对外展示 URL，优先 `/static/...`。
- `mime_type`：文件 MIME。
- `size_bytes`：文件大小。
- `checksum`：内容 hash，用于去重和引用检查。
- `width`、`height`、`duration_seconds`：媒体属性。
- `owner_user_id`：创建人。
- `project_id`：项目归属。
- `source_type`：upload / generated / imported / external / cache。
- `generation_id`、`task_id`：生成来源。
- `deleted_at`：软删除时间。

## 生命周期

```text
创建来源
  -> 上传 / 模型生成 / 导入 / 截帧 / 合成
  -> 写入文件
  -> 提取元数据
  -> 写数据库引用
  -> 业务对象引用
  -> 可选加入素材库
  -> 可选生成缩略图
  -> 被分镜/视频/导出引用
  -> 删除请求
  -> 引用检查
  -> 软删除或移动 trash
  -> 延迟物理清理
```

## 引用关系

删除文件前必须检查：

- 是否是角色、场景、道具主图。
- 是否在本剧库或公共素材库中。
- 是否被分镜作为参考图、首帧、尾帧、关键帧使用。
- 是否被视频生成任务引用。
- 是否属于已完成导出结果。
- 是否仍有任务正在处理。

有引用时不得直接物理删除。可以只解除当前业务对象引用，或把文件标记为待清理。

## 清理策略

- `temp/`：按任务结束时间或最后修改时间定期清理。
- `cache/`：可重建缓存，允许按容量或时间清理。
- `trash/`：保留固定周期后物理删除。
- 业务文件：只有无引用、已软删除、无任务占用时才能物理删除。
- 清理任务必须有 dry-run 模式，先输出将删除列表和原因。

## 验收清单

- 数据库不保存大段媒体正文。
- 删除素材不会误删仍被分镜、角色、视频引用的文件。
- 生成成功后的主展示地址优先为 `/static/...`。
- 部署不会覆盖服务器持久化文件。
- 清理脚本可以先预览，再执行。
