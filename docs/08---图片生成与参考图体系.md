# 图片生成与参考图体系

最近更新：2026-06-10

## 功能定位

图片生成负责把角色、场景、道具和分镜文本转为稳定视觉参考。参考图体系负责给视频生成提供视觉合同，尤其是角色一致性、场景空间和首尾帧连续性。

## 当前项目情况

- 图片生成记录落在 `image_generations`，包含项目、分镜、角色、场景、提示词、负面词、模型、AI 配置、帧类型、批次、锁定、选中、参考图和实际参数。
- 当前支持角色图、角色四视图、场景图、场景四视图、道具图、分镜图、首帧、尾帧和上传图片。
- 分镜图生成会服务后续图生视频；首尾帧用于视频连续性和镜头衔接。
- 已有图片列表、创建、上传、更新、删除接口，也支持按分集批量生成背景/分镜图。
- 图片接口规范由 AI 配置中的 `api_protocol` 决定协议大类：普通中转走 OpenAI 兼容，CLIProxyAPI 的 `gpt-image-2` 走专门兼容分支；前端不再展示单独的 OpenAI GPT Image 官方接口，后台仅保留旧配置兼容。
- 当前最需要补的是参考图优先级和“哪张图会被用于视频”的可见性。

## 当前已落地能力

- 支持角色图、四视图、场景图、道具图、分镜图生成。
- 支持分镜首帧、尾帧提示词生成和保存。
- 支持上传图片作为角色、场景、道具或分镜参考。
- 支持图片生成记录、选中、锁定、批次、参考图等元数据。
- 支持图片导入素材库。
- 支持 CLIProxyAPI `gpt-image-2` 适配：同一条配置可选择调用 `/images/generations|/images/edits`，也可选择调用 `/responses` 的 `image_generation` 工具；后端按 Codex/CLI 兼容方式少传易冲突字段，并解析 Images/Responses 两类流式事件。
- OpenAI GPT Image/CLIProxy 适配会按 `gpt-image-2` 官方尺寸约束归一化：边长不超过 3840、边长为 16px 倍数、长短边比例不超过 3:1、总像素不超过 8294400；小图会抬到最低有效像素，超大图会按比例压到有效范围。
- 图片返回格式已统一为“接口中间态可为 base64 / URL，本地保存成功后对外引用优先写 `/static/<local_path>`”。base64 不应长期作为前端主展示字段或参考图持久引用。
- 参考图入参会优先解析服务器本地文件：`/static/projects/...`、`projects/...`、localhost URL 和可识别的静态 URL 会先从 `data/storage` 读取，再以二进制/Blob 形式提交给支持图生图的接口，避免重复拼接 `/static/static/...`。
- 参考图解析和生成结果 URL 归一化已从 `imageClient.js` 抽到 `backend-node/src/services/ai-image`，后续服务商 adapter 可复用同一套文件/URL/base64 契约。
- 图片服务商协议已按 adapter 拆分：OpenAI 兼容/Seedream、OpenAI GPT Image/CLIProxy、DashScope/Qwen、Gemini、Kling、NanoBanana 分别维护请求构造、轮询或流式解析。
- OpenAI GPT Image/CLIProxy adapter 已开始抽取 `usage` 与轻量 `diagnostics`，并由图片路由门面写入脱敏模型调用日志。
- OpenAI GPT Image/CLIProxy adapter 已支持 Responses API 请求体构造：主模型用于 Responses 推理，图片工具默认 `gpt-image-2`，参考图会以 `input_image` 随同用户提示词提交。

## 当前问题与不足

- 不同参考图来源的优先级需要明确：手动上传、分镜图、首帧、尾帧、角色图、场景图、道具图。
- 图片生成后的“选中图”和“可用于视频图”需要更清晰。
- 分镜图质量与视频成败强相关，但目前缺少正式检查步骤。
- 需要持续记录图片生成耗时节点，区分慢在上游模型、代理/中转、参考图下载、响应读取还是本地存储。
- 2026-06-09 服务器排查显示，`gpt-image-2` 经 CLIProxyAPI 的 2K 图片任务主要慢在外部 API 等待阶段：首字节常见几十到两百秒，完整返回常见十几分钟；本地保存不是主要耗时。是否为中转站或上游模型导致，需要用同提示词、同尺寸、同参考图做直连/中转 A/B 测试。
- `gpt-image-2` 不支持透明背景请求，也不允许在编辑/参考图请求中修改 `input_fidelity`；适配层应跳过这类易报错字段。CLIProxyAPI 经实测可能支持 `quality`，但默认仍保守跳过，确认当前中转支持后可通过配置 `settings.image.send_quality=true` 打开发送。
- Responses API 与 Images API 的字段支持并不完全一致；当前 UI 只暴露 API 接口切换，`tool_model`、`send_quality`、`partial_images` 等仍保留为内部/高级 settings，避免普通配置页变成参数堆。
- 2026-06-10 排查发现，Responses API 流式图片请求可能先返回 SSE 响应头但长时间不结束；适配层必须把请求超时覆盖到响应体读取阶段，否则任务会一直停在 `processing`。

## 方案池 / 待确认讨论

- 建立参考图优先级矩阵，按经典模式、全能模式、首尾帧模式分别规定。
- 给分镜图增加质量标记：可用、需重画、已锁定。
- 给角色、场景、道具图增加“参与哪些分镜”的反向索引。
- 抽卡预览阶段可考虑使用更轻的图片规格，满意后再走高清正式图，避免每次都用高成本长耗时参数。
- `gpt-image-2` 默认并发不宜过高。图片路由策略默认并发为 4，最大限制为 8；批量抽卡或批量分镜图应先按小并发运行，再按接口实际耗时调整。前端后续应展示排队、首字节等待、模型返回和本地保存的分段诊断。
- 图片 adapter 主体已经拆出；下一步重点不再是搬文件，而是给每个接口规范补诊断摘要：请求 endpoint、参考图数量与格式、首字节耗时、总耗时、返回中间格式、是否成功本地落盘。
- 当前 usage/diagnostics 先覆盖 OpenAI GPT Image 与 CLIProxyAPI，其他图片 adapter 后续按同一返回契约补齐。
- 对 CLIProxyAPI 的 Images API 流式中间图保持“可用但不强依赖”的策略：`partial_images` 可为 0-3，但前端和任务状态不能假设实际返回数量固定。

## 待开发清单

- 明确批量生成分镜图前的依赖检查。
- 给图片生成失败提供更友好的模型配置提示。
- 前端后续可显示图片生成耗时诊断摘要，但不暴露 API Key、完整 prompt 或大段 base64。
- 建立图片生成链路的格式契约：模型返回格式、服务端中间态、`image_generations.image_url`、实体主图 `image_url`、`local_path` 和参考图输入格式都要在接口文档中明确。
- 给 DashScope、Gemini、Kling、NanoBanana adapter 补齐统一 `usage/diagnostics` 返回摘要。

## 验收清单

- 用户能明确知道哪张图会被用于视频生成。
- 手动选择或锁定的图片不会被批量生成误覆盖。
- 分镜图、首帧、尾帧与视频生成参数一致。
- 生成成功后，角色、场景、道具、分镜的主图展示地址优先为 `/static/...`，数据库中不应继续写入大段 base64 作为主展示字段。
- `/static/projects/...` 作为参考图再次送入图生图时，不会被拼成 `/static/static/projects/...`。
