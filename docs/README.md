# qnvideo-gzt 项目功能档案

最近更新：2026-06-09

本目录用于长期维护芊柠AI视频工作台的产品方向、功能设计、工程约束、讨论结论、开发清单和验收标准。

这里不是临时备忘录。凡是会影响后续开发判断的内容，都要落到对应文档中，避免方案只停留在聊天记录里。

## 维护规则

- 开始修改功能前，先阅读对应功能档案。
- 没有对应功能档案时，先创建文档，再进入开发。
- 讨论产生的新需求、新判断、新取舍，即使暂不开发，也要写入对应文档的“方案池 / 待确认讨论”。
- 开发完成后，同步更新“当前已落地能力”“当前问题与不足”“待开发清单”“验收清单”和“更新记录”。
- 涉及多个功能区的系统性决策，先写入 `00---产品总纲与创作流水线.md`，再在相关功能文档中补充细节。
- 文档里不得写入真实密码、API Key、代理订阅、token、数据库连接串等敏感信息。

## 当前项目功能地图

| 功能区 | 当前入口 | 当前状态 |
| --- | --- | --- |
| 项目管理 | `/`、`/drama/:id` | 已有项目列表、创建、导入、导出、删除、剧集信息和资源概览 |
| 制作工作台 | `/film/:id` | 已承载剧本、角色、场景、道具、分镜、图片、视频、合成和一键流程 |
| AI 配置 | `/ai-config` | 管理员功能，已支持模型配置、提示词覆盖、业务场景映射、SD2 资产管理 |
| 用户管理 | `/admin/users` | 管理员功能，已支持创建用户、启停用户、删除无项目用户 |
| 媒体素材库 | `/media-library` | 已支持图片/视频筛选、搜索、上传、预览、批量删除和创建人权限 |
| 数据底座 | PostgreSQL + `data/storage` | 结构化数据入库，媒体文件落盘，数据库保存引用 |
| 部署形态 | Docker Compose | Web 与 backend 容器部署，服务器持久化目录不可被覆盖 |

## 专业视角

每次重要功能讨论和开发，都应至少从以下视角过一遍：

- 产品总导演：这个功能是否服务完整创作链路，是否降低用户决策成本。
- 小说与编剧策划：故事、人物、冲突、节奏、对白、旁白是否可拆成镜头。
- 分镜导演与影视摄影：景别、机位、运镜、光线、时长、镜头衔接是否成立。
- AI 动漫与视觉美术导演：角色一致性、画风、服化道、场景空间、关键帧是否统一。
- AI 视频与剪辑后期：生成模型限制、首尾帧、动作连续、视频时长、合成节奏是否可控。
- AI 模型架构师：模型配置、提示词结构、路由、失败切换、成本与额度是否可靠。
- 数据架构师：数据库字段、文件引用、用户归属、迁移、兼容性、清理策略是否清晰。
- 全栈工程师：前后端状态、接口、错误处理、权限、组件边界是否可维护。
- 工程规范负责人：目录、命名、代码风格、注释、依赖、测试组织是否规范。
- DevOps 与安全负责人：部署、环境变量、日志、备份、回滚、敏感信息是否安全。
- QA 验收负责人：自动化测试、真实浏览器验证、边界场景、残余风险是否记录。
