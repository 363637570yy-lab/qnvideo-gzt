# 工程架构目录与代码规范

最近更新：2026-06-10

## 功能定位

本文是 qnvideo-gzt 的长期工程规范和目录蓝图，用于约束后续功能开发、渐进式拆分、接口契约、数据边界、测试验收和部署风险。

本规范先建立口径，不代表立即大范围重构。后续做某个功能时，才在该功能边界内按本文逐步落地。

## 执行原则

- 规范先定，执行跟随功能推进。
- 不为了目录整齐单独跨功能搬迁大量文件。
- 每次优化一个功能时，同步检查该功能涉及的页面、组件、接口、服务、数据库字段、文件存储、测试和文档。
- 页面保持壳层，复杂业务域优先进入 `features/<domain>`；域内按业务工作台聚合 Vue、composable 和局部工具；真正跨域复用的能力再进入顶层 `components`、`composables`、`utils`。
- 后端 `routes` 只做鉴权、参数校验和响应封装；业务编排、模型参数、文件处理、任务状态和数据访问下沉到 `services`。
- 新增逻辑不得继续堆到已经超长的页面或 service 中，除非是临时补丁，并且必须记录后续拆分点。
- 代码改动后必须同步更新对应功能文档；方案暂不开发时，写入对应文档的“方案池 / 待确认讨论”。

## 当前项目情况

- 项目已经从上游 LocalMiniDrama 演化为用户自有的 AI 短视频创作平台。
- 当前同时包含剧本、角色、场景、道具、分镜、图片、视频、AI 配置、任务队列、素材库、权限和部署等多条业务链路。
- 前端部分核心页面承担了过多状态和业务逻辑，后续需要按功能逐步拆出组件和 composable。
- 后端模型接入、文件存储、任务队列和生成服务已经开始拆分，但仍需继续把路由、模型运行时、文件生命周期和业务编排分层。
- PostgreSQL 是正式结构化业务主库；媒体文件、生成结果和上传文件统一走 `data/storage`，数据库只保存引用和元数据。

目标分层：

```text

    filmCreate/              创作页功能域，按工作台聚合
```

前端页面规则：

- `views` 不承载复杂生成流程和大段业务判断。
- `api` 不写 UI 状态，不直接操作 store。
- 制作页拆分以“业务工作台”为第一边界，不以零散 UI 控件为第一边界；剧本、角色、道具、场景、分镜、视频合成各自保持完整工作台语义。
- 第一层目录只按“工作台”和 `shared` 公共层划分；按钮、筛选框、局部表单、局部弹窗、列表卡片先归属到对应工作台内部，避免把创作链路拆成难以理解的小碎片。
- `shared` 只放真正跨多个工作台复用且语义稳定的公共能力，例如图片预览、规格弹窗、模型策略标签、工作流选择器、任务状态；不得为了看起来细而把每个按钮、输入框、小布局提前拆成公共组件。
- 同一业务工作台内部的局部弹窗、列表卡片、局部工具栏、局部状态适配优先就近放在该工作台目录；只有确认会被多个工作台长期复用的组件或工具，才提升到 `features/filmCreate/shared`、顶层 `components`、顶层 `composables` 或 `utils/filmCreate`。
- 工作台内部拆分必须服务“打开该文件夹就能理解这个工作台”：可保留 `Workbench.vue`、`Dialogs.vue`、`useXxx.js` 等少量主文件；只有当单个主文件继续承载多个清晰子域时，才在本工作台目录内继续拆出控制区、媒体选择、字段状态、视频流程等局部模块。
- `FilmCreate.vue` 长期目标是制作工作台壳层，只保留路由、项目级状态装配、顶部项目设置和一级标签页编排；剧本、角色、道具、场景、分镜、视频合成必须拆到 `features/filmCreate/workbenches/` 对应业务工作台目录。
- 当前 `FilmCreate.vue` 向子工作台传递大 `ctx` 属于过渡期桥接，不是长期架构目标；后续应逐步改成父页只装配项目级上下文，各工作台自己消费所属 composable、显式 props/emits 和少量共享能力。
- `<script setup>` 拆分时必须保持依赖初始化顺序：传入 composable 的 `ref`、`computed`、函数和状态必须先声明再传递。不要把后面才由另一个 composable 解构出的值提前传给当前 composable，否则生产包可能出现 `Cannot access before initialization`，真实浏览器会渲染空白路由。
- 制作页顶部固定保留项目整体设置和“本次生成模型策略”栏；模型策略栏只做本次生成的文本/图片/视频/音频路由覆盖选择，默认自动，不替代管理员 AI 配置。
- 超过 800 行的组件，新增逻辑优先拆到同功能组件或 composable。
- 超过 1500 行的组件，本功能改动必须记录拆分方案或暂缓理由。
- 超过 3000 行的组件，原则上不继续堆新主逻辑。

### FilmCreate 拆分蓝图

制作工作台后续按“顶部常驻 + 一级标签页”拆分，不再把所有功能纵向堆在一个页面中。拆分后的理解方式是：一个业务工作台就是一个相对完整的创作房间，工作台内部可以有自己的 Vue、弹窗、列表、局部工具栏、状态逻辑和小工具；只有多个工作台反复共用且含义一致的能力，才进入 `shared`。

目标结构只表达业务归属，不要求为了补齐文件名而提前拆分；未落地的示例文件只有在对应工作台真实需要复用时才创建。

拆分顺序：

1. 先把制作页现有组件和 composable 收拢到 `features/filmCreate`，让目录第一眼呈现剧本、角色、道具、场景、分镜、视频等业务工作台。
2. 每个工作台内部优先保留 `Workbench.vue`、`Dialogs.vue`、`useXxx.js` 这种可读文件组合，不提前拆成过细的按钮、卡片和表单目录。
3. 再按工作台下沉业务状态：先项目加载、AI 路由、工作流规范，再剧本/角色/道具/场景，最后分镜、一键流程和视频合成。
4. 单个工作台文件超过 800-1200 行时，再在该工作台文件夹内继续拆局部组件；局部组件只服务本工作台时仍留在本工作台目录。
5. 只有确认会被多个工作台长期复用的组件、状态或工具，才进入 `features/filmCreate/shared`。

一级标签顺序建议为：`剧本`、`角色`、`场景`、`道具`、`分镜`、`视频/合成`。如果用户后续坚持“角色、道具、场景”顺序，资源 tab 可调整，但拆分边界不变。

### 制作工作台懒加载与聚合接口

制作页性能优化不能只靠拆组件。后续 `/film/:id` 必须按“首屏轻摘要 + 标签页按需加载 + 后端聚合视图模型”的方式演进：

- 打开项目时只请求工作台摘要，不拉取全量剧本、资源、分镜、图片历史和视频历史。
- 摘要接口由后端聚合项目基础设置、一键动作状态、本次模型策略摘要、各标签计数和运行任务数。
- 用户第一次打开某个标签页时，再请求该标签页的完整视图数据。
- 已加载标签页在前端内存缓存；用户点击刷新、生成任务完成、项目设置变化或摘要版本变化时再刷新。
- 生成任务轮询按当前标签页收敛；离开标签页后暂停高频轮询，只保留低频摘要刷新。
- 后端返回缩略图和稳定文件引用，前端不得在列表中请求原图、base64 或完整视频正文。
- 后端 tab 接口返回前端可直接渲染的 view model，不让前端串联多个接口再自行拼装权限、任务、文件、模型路由和引用状态。
- `storyboards` 与 `video-compose` 工作台接口只返回分镜轻量字段、主图/首尾帧引用和视频/合成摘要；媒体历史、关键帧历史、base64 图片和完整视频正文必须通过专门媒体接口按需加载。

## 后端目录规范

目标分层：

```text
backend-node/src/
  routes/                    薄路由：鉴权、参数校验、响应封装
  services/
    ai-runtime/              模型选择、路由、额度、用量、健康状态
    ai-text/                 文本模型协议、调用和解析
    ai-image/                图片模型协议、参考图、落盘和结果适配
    ai-video/                视频模型协议、任务查询、视频落盘
    ai-audio/                TTS 和音频模型协议
    workflow-presets/        工作流规范、提示词渲染、版本快照
    writing/                 故事、剧本、小说导入、分集文本和编剧结构
    storyboards/             分镜生成、镜头字段、首尾帧、连续性
    resources/               角色、场景、道具、素材引用
    storage/                 路径、元数据、缩略图、引用、清理、回收站
    tasks/                   任务状态机、队列、取消、恢复
    auth/                    账号、权限、身份上下文
  db/                        数据库适配、schema、迁移辅助
  migrations/                数据库迁移
  utils/                     纯工具函数
  test/                      按模块组织的测试
```

后端规则：

- `routes` 不直接拼复杂 prompt、不直接落盘、不直接调多个模型服务。
- AI 协议差异放在对应 `ai-*` adapter 中，不散落在业务 service。
- 任务状态、模型调用、文件存储必须可追踪，必要时用 `trace_id` 串联。
- 文件生成必须先落盘，再把稳定的 `local_path` / `/static/...` 写回业务对象。
- 业务 service 可以编排多个模块，但不应承担底层协议和文件生命周期细节。

## API 与响应规范

- 接口字段命名保持一致，避免同一概念出现 `type`、`preset_type`、`mode` 混用。
- 新接口需要有明确的请求参数、响应结构、错误码和权限边界。
- AI 配置、工作流规范、素材、任务、分镜、文件引用都必须支持管理员和普通用户权限差异。
- 长耗时生成动作必须走任务或可查询状态，不做无法恢复的黑盒请求。
- 外部模型错误要保留可排查信息，但最终响应不得暴露 API Key、代理、完整密钥或敏感请求头。

## 数据库与字段规范

- PostgreSQL 是正式结构化业务主库。
- 历史 SQLite 只作为迁移和兼容来源，不作为正式业务库。
- 表结构新增必须通过 migration，不能只依赖运行时补列。
- 所有业务核心表应明确 `created_by` / `owner_user_id` / `project_id` / `deleted_at` 等归属和软删除字段。
- JSON 字段只存适合半结构化扩展的数据；稳定查询字段应提升为显式列。
- 版本化对象需要保存版本号或快照，保证历史任务可追溯。

## 文件存储边界

文件存储生命周期由 `docs/14---文件存储生命周期与清理.md` 统一约束。本文只保留工程边界：

- 数据库保存 `local_path`、展示 URL、MIME、大小、尺寸、来源、任务、项目、创建人和引用关系。
- `data/storage` 是持久化目录，部署、拉代码、容器重建不得覆盖。
- 清理必须先引用检查，再软删除或进入回收站；物理删除需要 dry-run。
