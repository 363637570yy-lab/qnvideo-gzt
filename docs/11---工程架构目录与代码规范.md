# 工程架构目录与代码规范

最近更新：2026-06-08

## 功能定位

本文档维护 qnvideo-gzt 的工程落地规则，覆盖项目目录、代码组织、接口契约、字段规范、测试结构、部署影响和回滚要求。

这部分不能被合并到“全栈工程视角”一句话里。项目要真正落地，必须让产品方案、数据结构、前端状态、后端接口、文件目录、测试和部署都能对齐。

## 当前项目情况

- 本项目当前是单仓库产品化项目，`origin` 指向用户仓库，`upstream` 仅用于参考上游。
- 前端技术栈：Vue 3、Vite、Element Plus、Pinia、Vue Router、Axios。
- 后端技术栈：Node.js、Express、PostgreSQL、better-sqlite3 兼容层、Multer、Sharp、FFmpeg、AI 服务商客户端。
- 数据库迁移位于 `backend-node/migrations`，当前已有 25 个迁移文件。
- 前端和后端都已有 `node --test` 测试入口，前端还需要 `npm run build` 验证。
- 当前工程最大风险是 `FilmCreate.vue` 超长、生成链路跨文件复杂、AI 服务商参数差异大。

## 当前目录结构

```text

```

## 代码分层约定

- `frontweb/src/views`：页面级入口，避免无限膨胀；复杂逻辑应沉到组件或 composable。
- `frontweb/src/components`：可复用 UI 和业务组件。
- `frontweb/src/composables`：页面内可拆分的状态和业务流程。
- `frontweb/src/api`：接口封装，页面不得散落硬编码 API 路径。
- `frontweb/src/utils`：通用工具，避免承载复杂业务状态。
- `backend-node/src/routes`：HTTP 路由和参数校验，不承载大块业务逻辑。
- `backend-node/src/services`：业务服务、AI 调用、文件处理、任务编排。
- `backend-node/src/db`：数据库连接、迁移、兼容层。
- `backend-node/migrations`：数据库结构演进，新增字段必须可重复迁移或兼容已有数据。
- `backend-node/test`、`frontweb/test`：与改动范围对应的自动化测试。

## 字段与接口契约

- 新增前端字段前，必须确认后端保存、数据库字段、导入导出和旧数据兼容。
- 新增后端字段前，必须确认前端默认值、空值行为、权限校验和接口返回。
- 媒体文件字段要区分公网 URL、本地相对路径、本地绝对路径和可代理路径。
- AI 生成参数要区分用户输入、项目默认、模型默认、实际提交参数和服务商返回参数。
- 删除字段或改名字段前，必须检查导入导出、迁移、测试和服务器现有数据。

## 代码规范

- 优先沿用现有框架和本项目已有写法。
- 不把大型新功能继续堆进单个超长页面；需要拆 composable、组件或服务。
- 路由层只做输入校验、权限入口和响应封装，复杂逻辑放 service。
- 生成链路必须记录错误，不能只在前端显示失败。
- 涉及权限、数据归属、文件删除、批量生成、部署配置的改动必须有明确测试或手动验收。
- 注释只解释复杂业务约束，不写“把 A 赋值给 B”这类空注释。

## 测试与验收

改动前后按影响范围运行：

```powershell
cd backend-node
npm test

cd ..\frontweb
npm test
npm run build
```

如涉及页面体验或生成链路，还需要真实浏览器检查。

## 部署与回滚

- 部署前必须检查本地和服务器 Git 状态。
- 不覆盖服务器 `data/` 持久化目录。
- 不使用 `git reset --hard upstream/main` 覆盖用户产品化改造。
- 修改远程文件前优先备份。
- 容器重建后必须检查 Web、API health、容器状态和关键页面。

## 当前问题与不足

- `frontweb/src/views/FilmCreate.vue` 承载过重，后续任何素材库、分镜、视频生成相关优化，都必须避免继续扩大这个文件。
- 角色、场景、道具已有部分 composable，但页面模板、生成按钮、上传、任务状态、入库逻辑仍和制作页强耦合。
- 分镜相关状态、关键帧、参考图、视频生成、批量任务和修复流水线都集中在同一页面，应按“导演台模块”逐步拆出。
- 后端 `videoClient.js`、`imageClient.js` 把多个服务商能力、参数映射、响应解析放在一起，后续加模型会越来越难控。
- `routes/storyboards.js` 超过 1000 行，路由层偏厚，应把分镜校验、媒体绑定、生成编排继续下沉。
- 部分业务字段的契约散落在页面、service、migration 中，需要文档化。
- AI 模型服务商差异对接口参数影响很大，需要能力矩阵支撑。

## 方案池 / 待确认讨论

- 将制作页拆为多个工作区或子组件，先从素材工作流配置和分镜相关逻辑开始抽离。
- 为分镜、图片、视频生成建立统一参数摘要对象，便于前端展示和后端记录。
- 为关键业务对象建立字段契约表：项目、剧集、资产、分镜、图片生成、视频生成、任务。
- 为人物、场景、道具、分镜建立工作流规范配置模块；操作员页面只保留单选规范、少量执行选项和生成动作。

## 渐进式优化路线

1. 素材库工作流配置优化时，同步拆分人物、场景、道具生成 UI 和生成参数组装。
2. 分镜功能优化时，同步拆分分镜导演台组件、关键帧条、参考图选择、视频参数弹窗和批量生成状态。
3. AI 配置优化时，同步拆分 `AIConfigContent.vue`，把模型路由、冷却、测试连接独立。
4. 图片/视频模型适配优化时，同步拆分 `imageClient.js` 和 `videoClient.js` 的服务商适配层。
5. 新增数据库字段时，同步补字段契约、迁移说明、默认值、旧数据兼容和测试。
