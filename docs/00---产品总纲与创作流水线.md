# 产品总纲与创作流水线

最近更新：2026-06-08

## 产品定位

芊柠AI视频工作台不是单点 AI 生成页面，而是面向短视频、短剧、AI 动漫和小说改编的创作工作台。

核心目标：

- 把故事、角色、场景、道具、分镜、图片、视频、合成导出串成可持续迭代的项目链路。
- 让用户能分阶段确认内容，避免一键到底后成本失控、质量失控。
- 让管理员能统一维护模型能力、成本、权限和服务稳定性。
- 让普通用户在隔离的数据空间内完成自己的项目。

## 当前项目情况

- 当前是基于 `LocalMiniDrama` 改造的服务器版 AI 视频工作台，已不是上游镜像项目。
- 前端使用 Vue 3、Vite、Element Plus、Pinia；后端使用 Node.js、Express、PostgreSQL。
- 当前主工作台集中在 `frontweb/src/views/FilmCreate.vue`，功能完整但页面过重。
- 当前已具备从剧本到分镜图、分镜视频、整集合成的端到端链路。
- 当前最大短板是产品工作流、分镜方法论、字段契约和工程拆分还没有完全稳定。

## 标准创作链路

1. 创建项目并设置整体规格：画幅、主画风、默认语言、默认每段时长。
2. 输入故事梗概，生成或导入剧本。
3. 按剧本提取角色、场景、道具，并生成或上传视觉素材。
4. 生成分镜脚本，用户确认镜头节奏、对白、旁白、角色、场景和道具引用。
5. 生成角色图、场景图、道具图和分镜关键帧。
6. 生成分镜视频，必要时使用首尾帧、连续帧或全能视频模式。
7. 合成整集视频，并把生成结果沉淀为可复用素材。

## 产品原则

- 项目整体设置影响后续默认行为，单次生成设置只影响当前动作。
- 剧本是内容源，资产是视觉源，分镜是导演调度源，视频生成是执行源。
- 分镜必须成为核心工作台，而不是剧本文本和视频生成之间的临时中转。
- 所有高成本生成动作前，应有可检查、可暂停、可重试、可回滚的中间状态。
- AI 自动生成必须允许用户改写，用户改写内容优先于再次自动推断。
- PostgreSQL 存结构化业务数据，媒体文件存 `data/storage`，数据库只保存引用。
- 文档、代码、测试和部署说明必须同步演进。

## 跨功能待确认

- 是否把“项目整体设置”提升为项目创建时的强引导，而不是制作页里的隐藏设置。
- 是否把当前超大的制作页拆成“剧本台 / 资产台 / 分镜导演台 / 生成台”几个工作区。
- 是否为分镜建立独立的“镜头质量检查”状态，例如待确认、已确认、需重写、已锁定。
- 是否为模型路由引入“按项目指定策略”和“按任务临时覆盖”两级配置。

## 验收清单

- 从创建项目到合成视频的链路能被文档完整描述。
- 任一功能优化都能找到对应档案和验收标准。
- 重大方案不会只存在于聊天记录或代码注释中。
