# 日志审计与可观测性

最近更新：2026-06-09

## 功能定位

日志审计与可观测性用于回答平台运行中的关键问题：谁操作了什么、哪个任务卡住了、哪个模型慢、哪个 API 配置失败、文件是否落盘、错误能否追踪。

日志不是简单打印到控制台，而是平台后台能力。当前模型调用日志已先接入 AI 配置“运行观测”页，完整操作审计和系统诊断页面仍属于后续能力。

## 日志分类

| 类型 | 作用 | 示例 |
| --- | --- | --- |
| 操作日志 | 记录用户行为和管理动作 | 登录、创建项目、删除素材、修改 AI 配置 |
| 任务日志 | 记录异步任务阶段和耗时 | 排队、开始、调用模型、下载文件、落盘、完成 |
| 模型调用日志 | 记录模型请求诊断和消耗 | 配置、模型、耗时、usage、错误类型 |
| 文件日志 | 记录上传、下载、删除、清理 | 上传原图、生成缩略图、孤儿文件清理 |
| 系统日志 | 记录服务和系统异常 | 服务启动、数据库错误、队列异常 |
| 安全日志 | 记录敏感和高风险操作 | 密码修改、管理员操作、权限拒绝 |

## 脱敏规则

- 不记录完整 API Key、token、代理订阅、数据库连接串。
- 不记录大段 base64；只记录长度、hash、mime、前缀摘要。
- Prompt 默认记录 hash、长度、业务对象引用；完整 prompt 是否保存必须可配置。
- 用户上传文件路径可记录相对路径，不记录本机绝对隐私路径。
- 最终回复和提交信息不得包含敏感日志内容。

## trace_id 规范

- 每个长耗时生成链路应有 `trace_id`。
- 同一个 `trace_id` 贯穿：前端触发、后端任务、模型调用、文件下载、数据库更新。
- 用户反馈“某个生成很慢”时，应能通过 `task_id` 或 `trace_id` 查到耗时阶段。
- 当前模型调用层已由 `modelCallLogger` 自动生成或沿用 `trace_id`，并同步写入 `ai_model_call_logs` 与 `ai_usage_events`。

## 前端页面规划

- 操作日志：按用户、动作、资源、时间筛选。
- 任务日志：按任务、项目、状态、错误类型筛选。
- 模型调用日志：按服务类型、模型、配置、耗时、状态筛选；当前已在 AI 配置“运行观测”页提供基础查询。
- 系统诊断：健康检查、队列状态、最近错误、存储状态。
- 日志详情：显示脱敏请求摘要、响应摘要、耗时分段和关联任务。

## 当前项目情况

- 当前已有普通日志输出和任务状态记录。
- 当前已新增轻量 `ai_model_call_logs`，文本、Vision、OpenAI GPT Image/CLIProxy 图片和 TTS 会开始写入脱敏调用摘要。
- 当前已有 AI 配置“运行观测”页，可查看模型调用摘要、配置/模型 Top 和最近脱敏调用日志。
- 当前模型调用日志与用量事件已具备 `trace_id` 字段，并支持按 `trace_id` 查询；完整前端触发、任务步骤、文件落盘的全链路 trace 仍待继续贯通。
- 当前模型运行查询已支持按 `task_id` / `project_id` 过滤，任务状态接口会返回关联模型调用诊断摘要，包含 trace、错误、耗时和用量。
- 当前尚无独立日志中心和操作审计查询。
- 图片生成已开始补耗时和响应格式诊断，后续应继续统一到模型调用日志。

## 待开发清单

- 建立统一 `traceContext` 和日志服务，把前端请求、异步任务、模型调用、文件落盘串成同一 trace。
- 为模型 adapter 统一记录请求耗时、首字节、总耗时、usage 和错误类型。
- 在现有运行观测基础上补前端任务诊断展示、详情跳转和首字节耗时。
- 为 AI 配置、用户管理、素材删除、项目删除等高风险动作补操作日志。
- 增加管理员日志查询页面。
- 制定日志保留周期和清理策略。

## 验收清单

- 管理员能查询某个任务为什么慢或失败。
- 能区分慢在排队、模型响应、参考图下载、本地落盘还是合成导出。
- 敏感信息不会出现在日志页面、最终回复、提交信息中。
- 普通用户只能看到自己项目相关任务日志，不能看到系统敏感日志。
